# Edge Cases — Digital Lending and Access

Digital lending introduces a second fulfillment layer on top of the physical circulation model: every checkout must negotiate a DRM token with a third-party provider (OverDrive), and every return must revoke that token before the license is released back to the pool. Failures in either step have direct impact on member access and license availability. This document catalogues the failure modes unique to that layer, their downstream effects, and the detection and recovery strategies each requires. All rules reference the canonical business-rule identifiers (BR-06, BR-10, BR-11) and map to concrete API endpoints, domain events, and `DigitalLoan` status transitions.

## Failure Mode Reference Table

| Failure Mode | Impact | Detection | Mitigation / Recovery |
|---|---|---|---|
| **DRM token issuance failure** — OverDrive API returns a non-2xx error or times out during the `GET /digital-loans/{id}/token` call after the loan record is created | Loan record sits in `PENDING_TOKEN` indefinitely; member cannot open the title; license is consumed but inaccessible | Loan remains in `PENDING_TOKEN` beyond the 60-second SLA threshold; a scheduled monitor queries `SELECT * FROM digital_loans WHERE status = 'PENDING_TOKEN' AND created_at < NOW() - INTERVAL '60 seconds'` and fires an alert | Retry token issuance with exponential back-off (1 s, 2 s, 4 s, max 3 attempts). If all retries fail, set loan status to `RETURN_FAILED`, release the license back to the pool via a compensating transaction, notify the member with `HTTP 502` and a user-facing message ("We couldn't activate your digital loan — please try again"), and emit `DRMTokenRevoked` with `reason=ISSUANCE_FAILURE`. Staff can force-retry from the admin panel, which re-enters the token issuance flow idempotently using the original `loanId`. |
| **Simultaneous digital loans exceeding the cap** — Member already holds 3 active digital loans (BR-06); a 4th `POST /digital-loans` request arrives | Without enforcement, the member would hold 4 concurrent digital loans, violating the policy cap and potentially consuming an extra license | Policy guard runs a `SELECT COUNT(*) FROM digital_loans WHERE member_id = ? AND status IN ('PENDING_TOKEN','ACTIVE','PENDING_RETURN')` before issuing any new loan | Return `HTTP 422` with policy code `DIGITAL_LOAN_LIMIT_EXCEEDED` and a body listing the 3 active loan IDs and their due dates. Do not create a loan record or consume a license. The member must return an active title before borrowing a new one. The API is idempotent: repeated 4th requests all yield the same 422 until the count drops. Log the event for abuse detection. |
| **Content provider API downtime** — OverDrive/Libby is completely unreachable for 2 or more hours | New digital loans cannot be activated; pending returns cannot be processed; members see activation errors across all new checkouts | The OverDrive API client's circuit breaker opens after 5 consecutive failures within a 30-second window. The health-check poller (`GET https://api.overdrive.com/healthcheck`, every 30 s) triggers a `DRMProviderDown` event and a PagerDuty P2 alert after 3 consecutive non-200 responses | While the circuit breaker is open, `POST /digital-loans` enqueues the loan request (status `PENDING_TOKEN`) and responds `HTTP 202 Accepted` with a `Retry-After` header. Members receive an email when the provider recovers and the loan is activated. Pending return requests are similarly queued. The circuit breaker transitions to half-open after 60 s and probes with a single request; on success it closes. All queued operations are drained in order. Emit `DRMProviderDown` on open and a recovery event on close. |
| **Digital item access after return** — Member attempts to open a title using a DRM token after staff or the auto-return job has recorded the return in the system | Member presents a locally cached token; the provider may still honour it until its TTL expires, giving unintended access beyond the loan window | The `POST /digital-loans/{id}/return` flow calls the OverDrive token revocation API synchronously before setting the loan to `RETURNED`. If revocation confirms success, the token is immediately invalid at the provider level | Always revoke the token with the provider before persisting the `RETURNED` status. Use a two-phase write: (1) call `DELETE /token/{drmTokenId}` on OverDrive; (2) only on HTTP 200 or 204 set `status = RETURNED`. If revocation fails, set `status = PENDING_RETURN` and retry every 15 min for up to 24 h. Emit `DRMTokenRevoked` on success. Members who attempt to open a revoked token will receive an access-denied error from the provider's reader app, which is the expected outcome. |
| **Patron device limit exceeded** — The DRM provider enforces a 2-device registration limit; the member has 3 devices registered at the point of access | The provider rejects the token activation on the third device; the member sees a cryptic DRM error with no library-branded guidance | OverDrive returns `HTTP 403` with error code `DEVICE_LIMIT_EXCEEDED` during token redemption on the member's device; the reader app surfaces a raw error unless the library has mapped it | The `GET /digital-loans/{id}/token` response includes a `deviceLimitExceeded: true` flag when the upstream 403 is detected, alongside a `manageDevicesUrl` pointing to the member's OverDrive account device-management page. The member-facing message reads: "You've reached the device limit for this DRM provider. Please remove a device from your OverDrive account to continue." No library-side data changes are required; this is a provider account constraint. Staff can document the resolution path in the help portal. |
| **Content format not supported** — The only available version of a title is a PDF, but the member's device or app cannot render PDF; the member expected an EPUB | Member downloads a file they cannot open; support tickets increase; license is consumed unproductively | Format metadata is stored in `digital_catalog.available_formats[]` at ingest time. The `POST /digital-loans` request body includes `preferredFormat`; the API validates availability before creating the loan | Before creating the loan record, validate that `preferredFormat` exists in `available_formats`. If not, return `HTTP 409` with code `FORMAT_NOT_AVAILABLE` and the list of supported formats. If the member proceeds with an unsupported format and later requests a return within 15 minutes of activation ("wrong format" return), staff can waive any policy restrictions and mark the return with reason `FORMAT_MISMATCH`. The license is released immediately. Log format mismatches to inform future acquisition decisions. |
| **License count exhaustion** — All 5 purchased licenses for a high-demand e-book are checked out; a new borrow request arrives | Member cannot borrow the title; if handled incorrectly, the system could oversell licenses and create orphaned loan records | License availability is tracked in `digital_license_pool(title_id, total_licenses, checked_out)` with a database-level check constraint `checked_out <= total_licenses` and a `SELECT ... FOR UPDATE` lock on the row during each loan transaction | `POST /digital-loans` acquires a row-level lock on the license pool entry, checks `total_licenses - checked_out > 0`, and proceeds only if true. On exhaustion, return `HTTP 409` with code `LICENSE_POOL_EXHAUSTED` and the current queue depth. Automatically place the member at the back of the waitlist (identical to the physical reservation flow) and notify them via email when a license becomes available. The waitlist position and estimated wait time are returned in the 409 body. |
| **Digital loan auto-return during active reading session** — The 14-day loan window (BR-10) expires at midnight while the member is actively reading mid-chapter in the provider's reader app | The auto-return job revokes the token and releases the license; the member's reading session is interrupted with an access-denied error from the provider | The auto-return scheduler runs every 15 minutes, querying `SELECT * FROM digital_loans WHERE status = 'ACTIVE' AND due_date <= NOW()`. Matching loans are transitioned to `PENDING_RETURN` before the revocation API call | The auto-return job emits `DigitalLoanAutoReturned` and sends a push notification plus email 60 minutes before the due date: "Your loan of *[Title]* expires in 1 hour." If the revocation API call fails, the job retries every 15 minutes for 24 hours. After 24 hours without a successful revocation, the loan is marked `RETURN_FAILED` and escalated to staff. The license is not released back to the pool until the status reaches `RETURNED`, preventing a double-issue. Members with `Scholar` or `Premium` membership receive an automatic 3-day grace-period renewal if a license is available and no one is on the waitlist. |
| **Content file corrupted at provider** — The member downloads the e-book file and the local integrity check reports a SHA-256 checksum mismatch against the value stored in `digital_catalog.file_checksum` | Member receives a broken or unreadable file; they may attempt repeated downloads, inflating bandwidth costs and support load | The provider SDK returns a checksum with each download; the library's proxy layer compares it against the stored value and records the result in `digital_download_log(loan_id, checksum_match, timestamp)` | On checksum failure, return `HTTP 502` with code `CONTENT_INTEGRITY_FAILURE`; do not deliver the file to the member. Automatically request a fresh copy from OverDrive using the re-download endpoint. If two consecutive re-download attempts fail the checksum, raise a `ContentIntegrityAlert` event, notify OverDrive's content support channel, and inform the member: "This title has a file quality issue — our team has been notified. Please check back in 24 hours." The license is not consumed, and the loan is held in `PENDING_TOKEN` until a valid file is confirmed. |
| **Member changes email after DRM token issued** — A token is bound to the member's email address (used as the OverDrive account identifier); the member updates their email in the library system; the old email may expire or be reassigned | The member can no longer redeem or manage the active token under their new email; they lose access to an in-progress loan | No automated detection exists at issuance time. The mismatch surfaces when the member reports an access error. A nightly reconciliation job cross-references `members.email` against `digital_loans.drm_email` for active loans | On a successful email update (`PATCH /members/{id}` with a new email), the system checks for active digital loans. If any exist, it calls OverDrive's account-link update API to re-associate the active tokens with the new email. If the update fails, the member is warned: "Your active digital loans are linked to your previous email. Contact the library to update access." Staff can trigger a forced token re-issuance from the admin panel, which revokes the old token and issues a new one against the new email without resetting the loan window. |
| **Interlibrary loan (ILL) for digital content** — Member requests a digital title that is not in the home library's license pool and is not available from OverDrive under the library's existing agreement (BR-11) | Member cannot access the title through the standard digital lending flow; without a clear ILL path, they receive a dead end | `POST /digital-loans` returns `HTTP 404` with code `TITLE_NOT_IN_LICENSE_POOL`. The response body includes an `illEligible` flag driven by the member's membership tier (Premium or Scholar only, per BR-11) | If `illEligible = true`, the member is offered an ILL request through `POST /ill-requests` with `format=DIGITAL`. The $3 ILL fee (BR-11) is charged to the member's account and held until fulfilment. The ILL coordinator contacts partner libraries or negotiates a single-title OverDrive licence. Fulfilment SLA is 5–10 business days; the member is notified by email at each status transition. If `illEligible = false` (Standard membership), the member is shown an upgrade prompt and a physical-copy availability check across the consortium. |

### DRM Token Lifecycle

A DRM token is the cryptographic credential issued by OverDrive that authorises a specific member-device pair to decrypt and read a protected content file. Every `DigitalLoan` record is associated with exactly one token, tracked in `digital_loans.drm_token_id`. Tokens move through the following states:

| State | Meaning |
|---|---|
| `ISSUED` | Token was successfully obtained from OverDrive and stored; the member has not yet redeemed it on a device. |
| `ACTIVE` | Token has been redeemed on at least one device; the member is actively reading. |
| `REVOKED` | The library explicitly called the OverDrive revocation API and received confirmation; the token is no longer valid at the provider. |
| `EXPIRED` | The 14-day loan window elapsed and the auto-return flow successfully revoked the token. This is the normal terminal state for an on-time return. |
| `FAILED` | Token issuance was attempted but the provider returned a non-retryable error (e.g., title removed from catalogue). The loan is cancelled and the license is released. |

**Auto-return flow (BR-10):** The scheduled auto-return job runs every 15 minutes. For each loan where `due_date <= NOW()` and `status = 'ACTIVE'`:

1. Set `status = 'PENDING_RETURN'` and record `return_initiated_at`.
2. Call `DELETE /token/{drm_token_id}` on OverDrive.
3. On success: set `status = 'RETURNED'`, set `token_state = 'EXPIRED'`, decrement `digital_license_pool.checked_out`, emit `DigitalLoanAutoReturned`.
4. On failure: leave `status = 'PENDING_RETURN'` and schedule a retry in 15 minutes.
5. After 24 hours of consecutive failures (`return_initiated_at < NOW() - INTERVAL '24 hours'`): set `status = 'RETURN_FAILED'`, emit a `ReturnFailedAlert` event, and create a staff task. The license is held as consumed until staff resolves the failure.

The `DigitalLoanAutoReturned` domain event payload includes `loanId`, `memberId`, `titleId`, `drmTokenId`, `returnedAt`, and `wasGracePeriodApplied`.

### License Pool Management

The library purchases a fixed number of simultaneous-use licences for each digital title from OverDrive. These are tracked in `digital_license_pool`:

```
digital_license_pool(title_id PK, total_licenses INT, checked_out INT,
                     license_expiry_date DATE, last_renewed_at TIMESTAMPTZ)
```

Each successful `POST /digital-loans` increments `checked_out` under a `SELECT ... FOR UPDATE` row lock; each successful return decrements it. The database enforces `checked_out >= 0` and `checked_out <= total_licenses` via check constraints. Attempts to borrow when `checked_out = total_licenses` are rejected before the OverDrive API is called, preventing token issuance for a title the library cannot legally lend.

**Waitlist behaviour:** When `LICENSE_POOL_EXHAUSTED` is returned, the member is automatically enrolled in a reservation queue (identical mechanism to physical holds): `digital_holds(id, title_id, member_id, queued_at, notified_at)`. When a return decrements `checked_out`, the system immediately checks for the head of the digital hold queue and initiates a new `POST /digital-loans` on that member's behalf, skipping directly to the token issuance step. The member receives an email and push notification: "A copy of *[Title]* is now available — your loan has been activated." The member has 72 hours to open the title before the loan is auto-cancelled and the next person in the queue is served.

**License expiry and renewal:** Each OverDrive licence agreement has an expiry date stored in `license_expiry_date`. A nightly job checks for licences expiring within 30 days and creates a renewal task for the acquisitions librarian. If a licence expires while active loans exist, the existing loans are honoured to the end of their 14-day window (provider-enforced); no new loans are permitted against an expired licence. After renewal is recorded (staff updates `total_licenses` and `license_expiry_date`), the borrowing path re-opens automatically.

### Content Provider Resilience Strategy

All outbound calls to the OverDrive API are routed through a dedicated `OverDriveApiClient` service that wraps each call with a circuit breaker (Resilience4j or equivalent):

- **Closed state (normal):** Requests pass through. Failure rate is tracked over a 30-second sliding window. If the failure rate exceeds 50% across at least 5 calls, the circuit opens.
- **Open state (provider down):** All calls immediately return a `ProviderUnavailableException` without hitting the network. `POST /digital-loans` responds `HTTP 202 Accepted`; the loan record is created in `PENDING_TOKEN` status with a `queued_at` timestamp. A background worker drains the queue when the circuit closes.
- **Half-open state (recovery probe):** After 60 seconds in the open state, a single probe request is allowed. On success, the circuit closes and normal operation resumes. On failure, the circuit reopens and the 60-second timer resets.

**Health-check monitoring:** A separate poller calls `GET https://api.overdrive.com/healthcheck` every 30 seconds. Responses are logged to `provider_health_log(checked_at, status_code, latency_ms)`. After 3 consecutive non-200 responses, a `DRMProviderDown` domain event is emitted and a PagerDuty P2 alert fires for the on-call engineer. When the health check recovers (first 200 after a down streak), a `DRMProviderRecovered` event is emitted and the alert is auto-resolved. The queue drain begins 5 seconds after recovery to avoid a thundering-herd burst against the just-recovered provider — requests are processed at a maximum rate of 10 per second with jitter.
