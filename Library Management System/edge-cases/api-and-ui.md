# Edge Cases — API and UI

This document catalogues failure modes that span the API layer and user-facing interfaces of the Library Management System. Each entry describes a scenario that can occur in production under realistic load and usage conditions, its observable impact, how it is detected, and the mitigation or recovery strategy implemented in the Spring Boot microservice stack backed by PostgreSQL/RDS, Elasticsearch 8.x, Redis 7.x, Kafka, and AWS API Gateway.

---

## Failure Mode Reference

| Failure Mode | Impact | Detection | Mitigation / Recovery |
|---|---|---|---|
| **Concurrent API requests for the last copy** — Two `POST /loans` requests arrive simultaneously for the same `BookCopy` | One request silently succeeds and the second creates a double-loan, leaving the copy in an inconsistent state or triggering a duplicate loan record | The second request raises a constraint violation (`409 Conflict`) visible in application logs; Datadog alert fires on loan-duplicate metric | Acquire a row-level `SELECT FOR UPDATE` lock on the `BookCopy` row inside the loan-creation transaction. The first transaction commits and sets `status = CHECKED_OUT`; the second reads the locked row, sees `CHECKED_OUT`, and returns `HTTP 409` with body `{"error":"COPY_UNAVAILABLE","copyId":"..."}`. Callers must supply `X-Idempotency-Key`; duplicate keys within a 24-hour window return the cached original response without re-executing. |
| **Search index out of sync with database** — Elasticsearch index is stale; shows an available item that is already checked out | Patron reserves an item displayed as available; the reservation confirmation page shows a misleading "Ready for pickup" message before the hold queue check corrects it | Patron-facing error on the hold-confirmation step; Kibana query showing `doc.status=AVAILABLE` where `pg.status=CHECKED_OUT` exceeds threshold alert | Kafka consumer (`loan-indexer`) processes `LoanCreated` and `LoanReturned` events and updates the Elasticsearch document within <2 seconds under normal load. On cache miss or stale-hit, the OPAC service performs a synchronous PostgreSQL lookup as the authoritative fallback before returning availability to the UI. A nightly full re-index job (`02:00 UTC`) restores index integrity; during re-index, search latency increases by 200–400 ms and the fallback path is automatically promoted. |
| **Session timeout during checkout** — Staff JWT expires mid-transaction at step 3 of a 5-step checkout flow | The checkout flow breaks partway through; if the request carrying the expired token reaches a microservice after the previous step already wrote data, a partial loan record can be created | API gateway returns `401 Unauthorized`; client-side interceptor detects `401` on any checkout step | Access tokens expire after 15 minutes. The Angular checkout UI silently calls `POST /auth/token/refresh` using the 7-day refresh token before any step that may run close to expiry. If the refresh also fails (refresh token revoked or expired), the UI pauses the flow, presents a re-authentication modal, and serialises the in-progress checkout state to `sessionStorage`. On re-login, the state is restored and the flow resumes from the last confirmed step. Partial loan records created before the `401` are rolled back via a compensating `DELETE /loans/{loanId}` call issued by the checkout orchestrator on re-entry. |
| **Bulk import memory overflow** — A 50,000-record CSV upload exhausts server heap memory | The import worker pod crashes with `OutOfMemoryError`; partially imported records are committed to the database, leaving the catalogue in an inconsistent state | Pod OOM kill event in CloudWatch; import job status stuck at `IN_PROGRESS` in the `import_jobs` table for more than 10 minutes triggers a dead-job alert | CSV uploads are not loaded into memory in full. The `POST /admin/catalogue/import` endpoint streams the multipart body through Apache Commons CSV in chunks of 500 rows. Each chunk is processed and committed independently within its own transaction. A `import_jobs` record tracks `total_rows`, `processed_rows`, and `failed_rows`. If the pod is killed mid-import, an ECS task restart resumes from the last committed checkpoint row stored in the job record. Import files larger than 10 MB are rejected at the API gateway with `413 Payload Too Large`; files between 5 MB and 10 MB trigger an async processing path that returns `202 Accepted` with a job ID. |
| **Rate limiting edge cases** — A library automation script triggers rate limiting; legitimate bulk operations are blocked | The automation script receives `429 Too Many Requests`; time-sensitive overnight batch processes (due-date reminders, overdue notifications) are delayed or fail silently | `429` response rate rising above 1% on the `/v1/` prefix in API Gateway access logs; CloudWatch alarm on `4XXError` count | Public API keys are limited to 100 req/min; authenticated staff API keys to 1,000 req/min. Automation scripts must use a dedicated service-account API key with a `service-automation` scope, which is exempt from the staff rate limit tier and governed by a separate quota of 5,000 req/min. The `Retry-After` header is always present on `429` responses. Batch operations exceeding per-minute limits should use the bulk endpoints (`POST /loans/bulk`, `POST /notifications/bulk`) which are processed asynchronously and are not rate-limited per record. Rate-limit counters use a sliding-window algorithm backed by Redis to avoid boundary bursts. |
| **Pagination cursor invalidation** — A client holds a cursor token; records are deleted between paginated requests | The client receives a `410 Gone` or silently skips records, producing an incomplete result set; audit exports may miss items | Client-side error handling sees unexpected gaps in sequential IDs; integration tests with known fixture data detect missing rows | Cursor tokens encode the `(created_at, id)` keyset pair, not an offset. When the client requests the next page, the query uses `WHERE (created_at, id) > (:lastCreatedAt, :lastId)` so deleted rows simply produce no match and are skipped cleanly without shifting offsets. If the cursor points to a record that was deleted and the keyset lookup returns zero rows, the API returns `{ "data": [], "next": null }` with a `X-Cursor-Stale: true` header, signalling the client to treat the result set as complete. Callers that require guaranteed completeness (audit exports) must use the `GET /reports/…?format=snapshot` endpoint, which materialises a point-in-time snapshot before iteration begins. |
| **Barcode scanner input injection** — A malformed barcode string contains an SQL injection attempt | An attacker attempts to manipulate loan records or extract patron data through scanner input fields at the circulation desk | WAF rule at AWS API Gateway blocks requests containing known SQL meta-characters; Spring `@Valid` rejects the request before it reaches the service layer | All barcode values are validated against the pattern `^[A-Z0-9\-]{6,20}$` using a `@BarcodeFormat` custom constraint annotation. Input is bound to a `String` field and passed to a JPA repository method using a named parameter (`WHERE bc.barcode = :barcode`), never string-concatenated into a query. The AWS WAF `SQLi` managed rule group provides an additional network-layer block. Invalid barcode input returns `HTTP 400` with `{"error":"INVALID_BARCODE_FORMAT"}` and is logged with the source IP and terminal ID for security audit. No stack trace is exposed to the client. |
| **API response field removal** — A downstream client breaks when a response field is removed in a minor version update | A patron-facing mobile app or third-party ILS integration that relies on a field removed without notice throws a `NullPointerException` or renders a blank screen | Consumer-driven contract tests (Pact) fail in CI before the change reaches staging; field-removal without a deprecation cycle violates the API design rules enforced by the pipeline gate | All response schema changes are governed by the versioning policy described in `detailed-design/api-design.md`. Fields are never removed in a minor version. A field slated for removal is first annotated with the `Deprecation` and `Sunset` response headers, giving consumers a minimum 6-month window. During that window the field is still populated. After sunset, the field is moved to the `/v2/` prefix. Pact contract tests run on every PR; a producer cannot merge a change that breaks a registered consumer contract. The API changelog is published at `GET /v1/changelog`. |
| **Long-running report query timeout** — A monthly circulation report query exceeds the 30-second API gateway timeout | The staff UI displays a generic "request failed" message; no report data is returned; the query may continue running on the database, consuming connections | AWS ALB returns `504 Gateway Timeout` after 29 seconds; the report service logs the query plan and elapsed time | Report queries that are known to exceed 10 seconds are executed asynchronously. `POST /v1/reports/circulation` accepts the request, stores the parameters in the `report_requests` table, and returns `202 Accepted` with a `Location: /v1/reports/{reportId}` header. The report worker executes the query against a PostgreSQL read replica with a `statement_timeout = 120s` setting. The staff UI polls `GET /v1/reports/{reportId}` every 5 seconds and renders the result when `status = READY`. If the read-replica query also times out, the worker retries once with a pre-aggregated materialised view (`mv_monthly_circulation`) that is refreshed nightly. The final fallback is a pre-generated static report from S3. |
| **OPAC search with special characters** — Search for "O'Brien" or "García" fails due to unescaped input | Patron receives zero results or a `500 Internal Server Error`; searches containing apostrophes, accented characters, or right-to-left scripts are silently broken | Zero-result rate for multi-character queries spikes above baseline in Kibana; `ElasticsearchStatusException` stack traces appear in application logs | Search terms are passed to the Elasticsearch `multi_match` query via the Java High-Level REST Client, which serialises the query as JSON — apostrophes and Unicode characters are encoded automatically and never interpolated into raw query strings. The `catalogue` index uses the `standard` analyser with `asciifolding` token filter, so "García" matches "garcia" and vice versa. The `icu_analysis` plugin normalises Unicode to NFC form on both ingest and query. Input length is capped at 512 characters at the controller layer; inputs exceeding this are truncated with a `X-Input-Truncated: true` response header. |
| **Stale cache serving wrong availability** — Redis cache TTL of 60 seconds causes `AVAILABLE` to be shown when the item is already checked out | A patron navigates to an item detail page and sees `AVAILABLE`, attempts to place a hold, and receives an error only at hold-creation time; trust in the OPAC is eroded | Hold-creation failure rate rises; patron support tickets reporting misleading availability; Datadog cache-hit ratio metric diverges from expected baseline | Item availability is cached in Redis with a 60-second TTL keyed by `availability:{copyId}`. On every `LoanCreated` and `LoanReturned` event, the Kafka consumer calls `DEL availability:{copyId}` to invalidate the cache entry immediately, so staleness is limited to edge cases where the event is delayed. The OPAC item-detail page displays a `Last updated: N seconds ago` indicator when serving a cached response, and a "Check live availability" button that bypasses Redis and queries PostgreSQL directly. Availability data returned by the API includes a `cacheHit` boolean and `cachedAt` ISO-8601 timestamp for client-side transparency. |

### Concurrency and Idempotency

The primary concurrency risk in the circulation layer is two loan-creation requests racing for the last available copy of a title. The system handles this with a row-level pessimistic lock: the loan-service issues `SELECT * FROM book_copy WHERE id = ? FOR UPDATE` inside the loan-creation transaction boundary, ensuring that only one transaction can read-and-mutate the copy row at a time. The second concurrent transaction blocks on the lock and, once acquired, reads `status = CHECKED_OUT` and returns `HTTP 409 Conflict` without creating a loan record.

All mutating endpoints in the loans and digital-loans domains accept an `X-Idempotency-Key` request header (UUID v4, required on `POST /v1/loans` and `POST /v1/digital-loans`). The idempotency store is a Redis hash with a 24-hour TTL keyed by `idempotency:{clientId}:{key}`. On receipt of a request, the service checks the store before executing. If the key exists and the previous execution succeeded, the original response body and status code are returned immediately without re-running the transaction. If the key exists but the previous execution is still in flight (`status = PROCESSING`), the service returns `HTTP 409` with `{"error":"REQUEST_IN_FLIGHT"}`. This protects against duplicate submissions caused by client retries on network timeout.

### Search Index Consistency

Elasticsearch index state is maintained by the `loan-indexer` Kafka consumer group. The consumer subscribes to the `loan-events` topic and processes `LoanCreated`, `LoanReturned`, `LoanRenewed`, and `HoldActivated` events, updating the corresponding document in the `catalogue-items` index via an `UpdateRequest` with `doc_as_upsert = true`. Under normal load the end-to-end lag from PostgreSQL commit to Elasticsearch document update is less than 2 seconds. During the nightly full re-index job (02:00–03:30 UTC), the consumer is paused, the index is rebuilt from the PostgreSQL `book_copy` and `loan` tables into an alias-swap pattern (`catalogue-items-v{n+1}`), and the alias is atomically swapped on completion. During re-index, Elasticsearch search latency increases by 200–400 ms; the OPAC service detects this via a `/_cat/indices` health check and automatically promotes the synchronous PostgreSQL availability lookup as the primary path rather than the cache.

On any Elasticsearch query that returns an item marked `AVAILABLE`, the service performs a conditional real-time PostgreSQL check when the Kafka consumer lag for that partition exceeds 5 seconds (surfaced via a Confluent metrics endpoint polled every 10 seconds). This ensures that high-lag conditions do not degrade the accuracy of availability data displayed to patrons.

### API Versioning and Backward Compatibility

All public and internal API routes are prefixed with `/v1/`. The versioning policy (documented in full in `detailed-design/api-design.md`) prohibits removing or renaming response fields, removing enum values, or changing field types within a major version. When a breaking change is necessary, the new behaviour is introduced under `/v2/` while `/v1/` continues to operate. Responses from endpoints under an active deprecation cycle carry two headers: `Deprecation: true` (RFC 8594) and `Sunset: <RFC 1123 date>`, where the sunset date is always at least 180 days from the deprecation announcement.

Consumer-driven contract testing is implemented using Pact. Each registered API consumer (OPAC Angular SPA, mobile app, third-party ILS connector) publishes a Pact file to the Pact Broker. The `loan-service` and `catalogue-service` CI pipelines run `pact:verify` on every PR; a build that breaks a consumer contract is blocked from merging regardless of unit-test status. New fields may be added to responses in minor versions without a version bump, provided they are additive and nullable; consumers are required to use tolerant-reader patterns and ignore unknown fields.

The machine-readable API changelog is available at `GET /v1/changelog` and returns a JSON array of change entries with `version`, `date`, `type` (`added`, `deprecated`, `removed`), and `description` fields, allowing automated tooling to detect incompatible changes before they affect integration environments.
