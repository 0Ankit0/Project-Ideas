# Edge Cases — Circulation and Overdues

This document catalogues the failure modes and boundary conditions that arise in the circulation and overdue-management subsystem of the Library Management System. Each entry describes a concrete scenario that has been observed in library operations or that follows directly from the interaction between business rules BR-01 through BR-14 and the loan lifecycle (statuses: ACTIVE → OVERDUE → RECALLED → RETURNED / LOST). Coverage includes concurrency hazards, fine-calculation boundaries, multi-branch workflows, and membership expiry edge cases. Implementation teams should treat every row as a test-ready specification that maps to at least one API endpoint, domain event, or database constraint.

## Circulation Failure Mode Matrix

| Failure Mode | Impact | Detection | Mitigation / Recovery |
|---|---|---|---|
| **Simultaneous checkout of last copy** — Two staff scan the same item barcode at the same millisecond via `POST /loans` | Two loan records are created for one physical copy, one of them referencing a BookCopy that is no longer AVAILABLE; one patron leaves with an item whose loan is effectively phantom | Database unique constraint on `BookCopy.status = CHECKED_OUT` will cause one INSERT to fail; application logs surface a `409 Conflict`; daily loan-count reconciliation report flags copies with multiple ACTIVE loans | Use `SELECT FOR UPDATE` on the `BookCopy` row inside the checkout transaction so only one request acquires the row lock. The second request receives `HTTP 409` with error code `CIRC-001` and the staff screen displays "Copy just checked out — please verify." The failed request must never partially commit; the loan row must not be written unless the copy row update succeeds atomically. Include an idempotency key header (`X-Idempotency-Key`) on `POST /loans` so a network retry does not create a second loan |
| **Return of already-returned item** — Barcode scanned twice; second scan hits a loan with status RETURNED | If processed naively, the system would create a second RETURNED event, corrupt the loan history, and potentially suppress a real missing-item alert | `POST /loans/{id}/return` checks current loan status before writing; a RETURNED loan triggers a guard clause returning `HTTP 409` with code `CIRC-002`; staff screen displays "Item already returned on {date}" | Return endpoint must be idempotent for the RETURNED terminal state: re-scanning a RETURNED item is a no-op that returns `200 OK` with the existing loan record and a `already_returned: true` flag. If no ACTIVE loan exists for the barcode at all, return `404` with `CIRC-003` and prompt staff to check claimed-return workflow |
| **Lost item found after replacement charged** — Member returns a copy 60 days after paying a replacement fee via `POST /loans/{id}/declare-lost` | Catalog now has an unexpected physical copy that may conflict with procurement records; the member has already paid replacement cost but the financial ledger must not be double-debited | Staff scan triggers a lookup against all LOST loans for that barcode; the match surfaces a `found_after_replacement` condition in the API response | Initiate a reversibility assessment workflow: if the replacement copy was already ordered and received, the original copy is catalogued as a surplus duplicate and the returning patron receives a credit note (not an automatic refund) pending manager approval. If no replacement was received, reverse the replacement charge, reactivate the copy as AVAILABLE, and emit `loan.found_after_loss` domain event. Audit trail must capture the original fine, replacement charge, and any credit or reversal with staff ID |
| **Fine calculation boundary (midnight)** — Item due at 23:59 local time, returned at 00:01 the following calendar day | A naive date-string comparison (`due_date < return_date`) incorrectly charges one day of fines even though the item was returned fewer than three minutes late; member disputes the charge | Staff or member submits a dispute via the fine management interface; the fine record references `due_at` and `returned_at` timestamps | Store `due_at` and `returned_at` as UTC epoch timestamps in the `loans` table, never as bare `DATE` columns. Fine accrual computes `seconds_overdue = returned_at_utc - due_at_utc`; if `seconds_overdue ≤ 0`, no fine is raised. Grace period logic (BR-13) is applied after this UTC comparison, not before. This eliminates DST and timezone ambiguity. Existing loans created before this fix must be migrated to include a `due_at` timestamp derived from `due_date || ' 23:59:59'` in the branch's configured timezone converted to UTC |
| **Recall during active loan** — System sends a recall notification; BR-09 resets `due_date` to `today + 3 days` and sets loan status to RECALLED | Member may not see the recall notification in time; original due date may be weeks away and the new date is a surprise; failure to return by the recall deadline incurs RECALLED-rate fines at $0.50/day | `loan.recalled` domain event is emitted; notification service sends email + push within 60 seconds; if `returned_at` is null after `recall_due_at`, the nightly overdue job transitions the loan to OVERDUE with the recall fine rate | Recall must be atomic: update `loans.due_at`, `loans.status = RECALLED`, and emit `loan.recalled` in the same database transaction. If the member already returned the item before the recall was processed (race condition), the recall must be voided rather than applied. The staff screen must clearly distinguish RECALLED loans from standard OVERDUE loans so the correct fine rate ($0.50/day) is applied. A second recall on an already-RECALLED loan is a no-op |
| **Patron with borrowing block tries checkout** — Outstanding fines exceed $25 (BR-08); staff attempts a supervisor override at the desk | If override is allowed without audit trail, fine policy is undermined; if override is never allowed, edge cases such as reserved course-reserve items cause operational disruption | `POST /loans` returns `HTTP 403` with code `CIRC-008` and a `block_reason: FINE_THRESHOLD` payload; the checkout screen renders a hard-block banner rather than a soft warning | Override requires a staff member with role `CIRCULATION_SUPERVISOR` to supply their credential and a mandatory reason code (e.g., `COURSE_RESERVE`, `ACCESSIBILITY_NEED`). The override is logged to the `loan_overrides` audit table with `staff_id`, `reason_code`, `override_at`, and the patron's outstanding balance at the time. Overrides do not waive the fine; the balance remains and checkout proceeds with a flag `override_applied: true` on the loan record. More than three supervisor overrides for the same patron in 30 days triggers an alert to the branch manager |
| **Copy status inconsistency** — `BookCopy.status = AVAILABLE` in the database but the physical item cannot be located on the shelf | A patron reserves or is allocated a copy that staff cannot retrieve; reservation queue stalls; patron experience degrades | Staff perform a physical search and flag the item missing via the `PATCH /copies/{id}` endpoint with `status: MISSING`; the nightly inventory reconciliation report surfaces copies that have been AVAILABLE for more than 90 days without a checkout | When a copy cannot be located for a hold pickup, staff must set status to MISSING rather than leaving it AVAILABLE. The reservation subsystem must re-allocate the next available copy or requeue the hold. If no other copy exists in the system, the patron is notified with an estimated re-availability date. MISSING copies are automatically promoted to WITHDRAWN status after 180 days without a physical scan, subject to manager confirmation |
| **Renewal during recall** — Member calls `POST /loans/{id}/renew` for an item that has an active reservation or has been set to RECALLED status | Renewal would extend the loan, directly violating the recall agreement and delaying availability for the reserving patron; if the API silently succeeds, the recall is effectively nullified | `POST /loans/{id}/renew` checks for `loans.status = RECALLED` and for any `reservations` row with `status = ACTIVE` referencing the same `catalogue_item_id` before processing the renewal | Renewal must be blocked with `HTTP 409` and code `CIRC-003` if either condition is true. The response body must include `block_reason` (either `ITEM_RECALLED` or `ACTIVE_RESERVATION`) and the recall due date so the member understands their deadline. BR-03 explicitly states that active reservations block renewal; this check is enforced at the service layer, not just the UI. Maximum two renewals per loan (BR-03) is a separate counter check applied before the recall/reservation check |
| **Multi-branch return** — Item checked out at Branch A is returned at Branch B | `BookCopy.home_branch_id` does not match the receiving branch; the copy must be physically transported back or reassigned; the loan must be closed at the correct branch for statistical reporting | `POST /loans/{id}/return` at Branch B succeeds (the loan is closed) but sets `BookCopy.status = IN_TRANSIT` and `BookCopy.current_branch_id = BRANCH_B`; a `copy.in_transit` domain event is emitted | The return is accepted and the loan is closed immediately so the member is not charged additional fines while the copy is in transit. The copy status is set to IN_TRANSIT and the home branch is notified via the `copy.in_transit` event to schedule courier pickup. The copy must not be available for checkout at Branch B while IN_TRANSIT. Once the home branch scans the copy, it transitions back to AVAILABLE and `current_branch_id` is restored. Cross-branch return statistics are recorded for cost allocation |
| **Item scanned but checkout transaction fails** — Barcode read succeeds at the desk terminal, but the database write to `loans` fails mid-transaction (network partition or DB timeout) | The patron leaves with the item believing the checkout succeeded; the system has no record of the loan; the item appears AVAILABLE in the database while physically absent | The `POST /loans` request returns a `5xx` error or times out; the staff terminal displays an error banner; the patron's checkout receipt is not generated | The checkout endpoint must be wrapped in a single ACID transaction: `loans` INSERT and `BookCopy.status` UPDATE must both succeed or both roll back. The idempotency key on `POST /loans` ensures a retry from the terminal creates the loan exactly once even after a timeout. Staff must verify the checkout receipt before releasing the item; if no receipt is generated, the barcode must be re-scanned to confirm status. A dead-letter queue captures failed loan events for manual reconciliation within 15 minutes |
| **Membership expiry during active loan** — Member's membership expires while 3 books are checked out (e.g., annual membership lapses) | Expired members cannot borrow new items, but punishing them with immediate fines on already-checked-out loans is incorrect; however, unlimited grace causes abuse | The membership expiry job emits a `membership.expired` event; the loan service subscribes and marks all ACTIVE loans with a `membership_expiry_flag: true` without changing due dates | BR-14 defines a grace window (14 days by default, configurable per membership tier) during which existing loans continue on their original due dates and fine rules. After the grace window, overdue loans accrue fines normally. New checkouts are blocked immediately upon expiry regardless of the grace window. A renewal prompt is sent on expiry day and at day 7 of the grace window. If the member renews their membership within the grace window, the flag is cleared and no special action is needed |
| **Damaged item returned** — Member returns a water-damaged book; fine assessment must distinguish cosmetic blemishes from functional damage from total destruction | Flat replacement-cost charges for minor damage are unfair and drive disputes; no charge for severe damage is a financial loss; inconsistent assessment across staff creates liability | Staff flag the item at the return desk using the `PATCH /copies/{id}` endpoint with `status: DAMAGED` and a `damage_tier` field; a `copy.damaged` event triggers the adjudication workflow | Damage is assessed in three tiers (see Damaged Item Adjudication Workflow section below). The fine is created via the fine management service and attached to the loan, not the copy, so the financial record survives even if the copy is later withdrawn. The member is notified of the damage fine with a description of the assessment and the appeal process. Staff cannot waive damage fines exceeding $20 without supervisor approval, logged to the audit table |

## Fine Calculation Reference

Fines are calculated using the formula `days_overdue × daily_rate`, where `days_overdue` is derived from the UTC timestamp difference between `loans.due_at` and `loans.returned_at` (or `NOW()` for still-active overdue loans), divided by 86 400 seconds and rounded up to the nearest whole day. The daily rate is material-type-dependent per BR-04: Books at $0.25/day, DVDs at $1.00/day, Periodicals at $0.10/day, and Reserve items at $0.50/day. The total fine for any single loan is capped at three times the item's replacement cost, also per BR-04. Once the cap is reached, the loan status may be promoted to LOST via the `POST /loans/{id}/declare-lost` endpoint, which transfers liability from an accruing fine to a fixed replacement charge.

Grace period logic (BR-13) is applied as a post-calculation adjustment, not a due-date extension. Each material type carries a configured grace period (e.g., Books: 2 days, DVDs: 0 days, Periodicals: 1 day). If `days_overdue` is less than or equal to the grace period for that material type, the calculated fine is set to zero and no fine record is created. This prevents penalising members for returns that are marginally late due to library hours or transit. Grace period configuration is stored in the `material_type_policies` table and is versioned; a change to the grace period does not retroactively affect fines already assessed.

Midnight boundary handling is the most common source of fine disputes. The system must compare `due_at` and `returned_at` as UTC epoch integers. An item due at 23:59 local time is stored as its UTC equivalent, e.g., `due_at = 1700000399` (UTC). A scan at 00:01 the next calendar day in the same local timezone yields `returned_at = 1700000460`. The difference is 61 seconds — well within any grace period — and no fine is raised. Date-string comparison (`due_date < DATE(returned_at)`) would instead compute a one-day overdue and incorrectly raise a $0.25 fine. All timestamp storage must use integer UTC epoch values; display formatting to local time occurs only at the presentation layer.

## Simultaneous Checkout Concurrency

The `BookCopy` table uses a pessimistic row-level lock via `SELECT FOR UPDATE` during the checkout critical section. When `POST /loans` is received, the application begins a database transaction, executes `SELECT id, status FROM book_copies WHERE id = :copy_id FOR UPDATE`, and verifies that `status = AVAILABLE`. If another transaction holds the lock (i.e., a concurrent checkout of the same copy is in flight), the second transaction blocks until the first commits or rolls back, then re-reads the status. Since the first transaction will have set `status = CHECKED_OUT` before committing, the second transaction's status check will fail with `AVAILABLE` no longer being true, and it will roll back and return `HTTP 409` to the caller.

Optimistic locking is used as a secondary safeguard at the application layer for bulk operations (e.g., batch pre-checkouts for classroom reserves). The `BookCopy` row includes an `updated_at` timestamp; the application reads the row, holds the timestamp, and issues an `UPDATE ... WHERE id = :id AND updated_at = :read_timestamp`. A row-count of zero indicates a concurrent modification and triggers a retry with back-off up to three attempts before surfacing an error to the caller.

The `POST /loans` endpoint accepts an `X-Idempotency-Key` header (UUID, required for all terminal/POS clients). The idempotency store (backed by a Redis key with a 24-hour TTL) maps the key to the loan ID and response status. If the same key is received while the first request is still in flight, the second request blocks on the idempotency store entry rather than executing a second database transaction. If the key is received after the first request completed, the stored response is returned verbatim. This ensures that a network timeout followed by a retry from the desk terminal produces exactly one loan record and exactly one `BookCopy` status update.

## Damaged Item Adjudication Workflow

Damage is classified into three tiers that govern both the fine amount and the disposition of the physical copy:

**Tier 1 — Cosmetic damage** covers surface marks, minor cover scuffs, and page yellowing that do not impair readability. The copy remains in circulation. The fine is assessed at 10% of the item's replacement cost, with a minimum of $1.00 and a maximum of $10.00. No replacement order is triggered.

**Tier 2 — Functional damage** covers torn or missing pages, water damage that obscures text, broken DVD cases with a scratched disc, or binding failure. The copy is withdrawn from circulation and set to `BookCopy.status = WITHDRAWN`. The fine is assessed at 50% of the replacement cost. If the copy is the only copy of that title in the system, a replacement order is automatically raised in the acquisitions module via the `POST /acquisitions/orders` endpoint with `reason: DAMAGE_REPLACEMENT`.

**Tier 3 — Destroyed** covers items that are unserviceable: completely water-soaked, burned, or structurally destroyed. The copy is withdrawn and the fine equals 100% of the replacement cost (equivalent to the lost-item replacement charge). This tier follows the same financial pathway as `POST /loans/{id}/declare-lost`.

The staff workflow proceeds as follows: at the return desk, staff select the damage flag from the return screen, choose a tier using the damage assessment UI, and optionally attach a photo reference ID. The `PATCH /copies/{id}` call sets `status = DAMAGED` and `damage_tier = {1|2|3}`. A `copy.damaged` domain event is emitted, which the fine service consumes to create a fine record attached to the loan with `fine_type = DAMAGE` and the tier-appropriate amount. Simultaneously, the notification service consumes the event and sends the member an itemised damage notice with the fine amount, tier description, and appeal instructions. Appeals are handled via the `POST /fines/{id}/dispute` endpoint, which routes to a supervisor review queue. Staff cannot finalise a Tier 3 fine without a second confirmation from a `CIRCULATION_SUPERVISOR` role, and all damage fines above $20 are logged to the `loan_overrides` audit table regardless of role.
