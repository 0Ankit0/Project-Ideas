# Edge Cases — Catalog and Metadata

This document enumerates production-relevant failure modes for the Library Management System's catalog and metadata subsystem. It covers the `CatalogItem` and `BookCopy` domain entities, the `POST /catalog/items`, `POST /catalog/import`, `GET /catalog/items`, and `GET /catalog/items/{id}/copies` endpoints, and all ISBN lookup integrations (OpenLibrary, Google Books). Each scenario is described with its operational impact, detection mechanism, and a concrete mitigation or recovery path. Teams should use this document to drive test case design, runbook authoring, and constraint implementation.

## Failure Mode Reference Table

| Failure Mode | Impact | Detection | Mitigation / Recovery |
|---|---|---|---|
| **Duplicate ISBN detection** — Two `CatalogItem` records share the same ISBN | Search surfaces two competing records for the same title; hold queues and availability counts split across both records, misleading patrons and confusing staff during check-in/check-out | `UNIQUE` constraint on `catalog_items.isbn` raises a database integrity error; `POST /catalog/items` returns `409 Conflict` with `{"code":"DUPLICATE_ISBN","existingId":"<id>"}` | BR-15 enforces ISBN uniqueness at the API layer before the INSERT is attempted. When a duplicate is detected, the system presents a staff-facing merge workflow: the operator reviews both records side-by-side, selects the canonical record, and migrates all `BookCopy` rows and active holds to it before soft-deleting the duplicate. All merge actions are written to the audit log with the acting staff member's ID. |
| **Malformed metadata from ISBN API** — OpenLibrary returns incomplete or invalid JSON | `CatalogItem` is created with null or empty `title`, `author`, or `deweyClass` fields; patron-facing catalog displays blank cards; downstream search indexing may fail or produce zero-result queries | HTTP client catches `JsonParseException` and `JsonMappingException`; a structured validation step checks that required fields (`title`, `author`) are non-null and non-empty after deserialization; failures are logged at `WARN` level with the raw response body truncated to 2 KB | The enrichment pipeline falls back to the Google Books API when OpenLibrary returns an unparseable or structurally invalid response (see Metadata Enrichment Fallback Strategy below). If both APIs fail validation, the item is created in a `PENDING_METADATA` state and queued for manual entry by cataloging staff. A `catalog.metadata.enrichment.failed` domain event is emitted so the operations dashboard can surface incomplete records. |
| **Missing Dewey classification** — A new book has no DDC number assigned | Shelfing guides and physical location lookups break; browse-by-subject features return incomplete result sets; patron self-service kiosk cannot direct the patron to the correct aisle | `deweyClass` is nullable in the schema; the catalog API returns the field as `null`; a daily reconciliation job queries `SELECT id FROM catalog_items WHERE dewey_class IS NULL` and produces a report for cataloging staff | Staff are alerted via the operations dashboard and must assign a `deweyClass` before the item's copies can be moved to `AVAILABLE` status. The `BookCopy.status` is held at `IN_PROCESSING` until the parent `CatalogItem.deweyClass` is populated. A `catalog.dewey.missing` event triggers an automated suggestion using the Library of Congress Classification API if the title and author fields are present. |
| **ISBN lookup API downtime** — Both OpenLibrary and Google Books are unreachable | New item creation via ISBN lookup is blocked; staff cannot auto-populate title, author, publisher, or cover URL; catalog throughput drops to zero if staff rely solely on auto-enrichment | HTTP client enforces a 5-second connect timeout and a 10-second read timeout per attempt; circuit breaker trips after 5 consecutive failures within a 60-second window; a health check endpoint (`GET /catalog/health`) exposes API reachability as a dependency flag | The circuit breaker transitions the metadata enrichment pipeline to `MANUAL_ENTRY` mode, surfacing a prominent banner in the cataloging UI. Staff can create a `CatalogItem` by supplying all required fields manually; the `isbn` field is still validated for check-digit correctness locally. When the circuit closes again, any items in `PENDING_METADATA` state are automatically retried against the live APIs. Retry intervals follow exponential backoff: 30 s, 60 s, 120 s, 300 s. |
| **Bulk import partial failure** — A 500-record CSV import fails at row 347 | 346 records are committed while 154 are not; the catalog is in an inconsistent state relative to the source CSV; re-running the full import risks creating 346 duplicates | `POST /catalog/import` returns a `207 Multi-Status` body containing `{"processed":346,"failed":154,"firstFailureRow":347,"jobId":"<uuid>","failureDetails":[…]}`; the import job status is queryable at `GET /catalog/import/jobs/{jobId}` | The import pipeline processes records in transactional chunks of 50 rows. If a chunk fails, that entire chunk is rolled back and the job advances to `PARTIAL_FAILURE` state. The `jobId` and the last successfully committed chunk index are persisted, enabling staff to re-run the import from the checkpoint row (`POST /catalog/import` with `{"resumeFromRow":347,"jobId":"<uuid>"}`). Each row is validated for ISBN check-digit correctness and mandatory field presence before the chunk transaction opens. |
| **Item record with no physical copies** — A `CatalogItem` exists but its `BookCopy` count is zero | Patron searches find the title but availability always shows "No copies available"; hold requests are accepted but can never be fulfilled; the hold queue grows indefinitely | `GET /catalog/items/{id}/copies` returns an empty array; a nightly reconciliation job flags catalog items where `COUNT(book_copies) = 0` and the item is older than 48 hours; the operations dashboard surfaces these as "ghost records" | Items with zero copies are tagged with `availabilityStatus: NO_COPIES` in search index documents so patrons see an accurate status rather than a misleading "available" indicator. The hold subsystem rejects new hold requests with `409 {"code":"NO_COPIES_AVAILABLE"}`. Staff receive a dashboard alert to either create `BookCopy` records via `POST /catalog/items/{id}/copies` or suppress the item from discovery. If the item was created as a pre-order placeholder, it remains visible with an `EXPECTED` status until copies arrive. |
| **ISBN collision (different editions)** — An ISBN has been reused across different editions or regional printings | Patrons checking out a record may receive a physically different edition than expected; call number labels and cover images may not match the physical item; inter-library loan routing can fail if the wrong edition is dispatched | During `POST /catalog/items`, the system checks whether an existing record with the same ISBN has a different `title` substring or `author` value beyond a configurable Levenshtein distance threshold; a `WARN`-level `catalog.isbn.collision_suspected` event is emitted | Staff receive a collision warning before the record is saved, showing the conflicting existing record side-by-side. They can proceed by appending an edition qualifier to the title (e.g., "The Hobbit [UK 2nd ed.]") and recording the edition discrepancy in the `metadata` JSON field under `editionNote`. The ISBN is stored as-is to preserve lookup compatibility, and the `metadata.isbnCollision` flag is set to `true` for downstream audit and reporting. |
| **Author name variant** — The same physical author is stored as "Tolkien, J.R.R." in one record and "J. R. R. Tolkien" in another | Author-browse and faceted search return split result sets; patron searches for the author's complete works return incomplete lists; circulation reports double-count author entries | A fuzzy-match step in the cataloging pipeline computes normalized forms of incoming author strings (strip punctuation, lowercase, reorder surname/given) and compares them against existing authority entries; a similarity score above 0.85 triggers a `catalog.author.variant_detected` event | The catalog maintains an author authority table with a canonical form and a list of known variant strings. When a variant is detected, the incoming record is linked to the canonical authority entry rather than creating a new free-text author value. Staff can promote a variant to canonical via the authority management UI, which triggers a bulk re-index of all affected `CatalogItem` records. Authority records reference the Library of Congress Name Authority File (LCNAF) identifier where available. |
| **Cover image URL broken** — `CatalogItem.coverImageUrl` returns HTTP 404 at display time | Patron-facing catalog renders a broken image icon; mobile app layout shifts; accessibility screen readers announce a missing image without a meaningful fallback label | The frontend image component uses an `onError` handler to detect failed image loads; a background job issues HEAD requests against all stored `coverImageUrl` values nightly and records response codes in `catalog_items.cover_image_status` | On 404 detection, the system replaces the broken URL with a branch-configured generic cover placeholder served from internal CDN. A `catalog.cover.broken` event is emitted, and the enrichment pipeline schedules a single re-fetch from OpenLibrary and Google Books within the next enrichment cycle. If a new URL is found, it is written back to `CatalogItem.coverImageUrl` and the `cover_image_status` is reset to `OK`. |
| **Conflicting metadata from two API sources** — OpenLibrary reports publication year 1984; Google Books reports 1985 | Incorrect publication year propagates to catalog records, patron displays, and export feeds; bibliographic citations generated from the catalog contain wrong data; de-duplication logic that uses pub year as a signal may misclassify records | The enrichment pipeline compares scalar fields from both API responses and flags any field where the two sources differ; conflicting fields are written to `CatalogItem.metadata.conflicts` as a JSON array of `{"field","openLibraryValue","googleBooksValue"}` objects | OpenLibrary is treated as the higher-priority source for publication year, publisher, and DDC classification. When a conflict exists, the OpenLibrary value is used to populate the canonical field and the Google Books value is preserved in `metadata.conflicts` for staff review. Records with unresolved conflicts are surfaced in the cataloging dashboard under "Needs Review." Staff can accept either value or supply a manual override, which sets `metadata.manuallyVerified: true` and suppresses further automated conflict overwriting. |
| **Unicode in metadata fields** — An Arabic or Chinese title causes encoding issues in storage or display | Title, author, or subject fields are stored as garbled mojibake; search tokenization fails for non-Latin scripts; CSV exports to legacy ILS systems corrupt non-ASCII characters | Database collation is set to `utf8mb4_unicode_ci`; the API layer validates that incoming strings are valid UTF-8 before persistence; integration tests include fixture records with Arabic, Chinese, and Cyrillic titles; mojibake is detectable via a regex that flags replacement characters (`\uFFFD`) in stored strings | All text fields in `CatalogItem` are stored in `utf8mb4` columns. The HTTP API enforces `Content-Type: application/json; charset=utf-8` on all endpoints. CSV export uses UTF-8 BOM (`0xEF 0xBB 0xBF`) to ensure Excel compatibility. For legacy ILS integrations that cannot accept Unicode, a transliteration adapter (using ICU4J) is applied at the export boundary and the original Unicode value is preserved in the primary store. |
| **Mass deaccession with active loans** — Staff attempt to withdraw a batch of items where one or more copies have `status = CHECKED_OUT` | Checked-out copies are marked `WITHDRAWN` while still in a patron's possession; the loan record references a copy that is no longer in the active catalog; return processing breaks because the barcode resolves to a withdrawn item | `PATCH /catalog/items/{id}/copies/{copyId}` validates that `status != CHECKED_OUT` before allowing a transition to `WITHDRAWN`; the deaccession batch endpoint returns `422 {"code":"ACTIVE_LOAN_CONFLICT","affectedCopyIds":[…]}` for any copy with an open loan | The deaccession pipeline skips copies with `status = CHECKED_OUT` and logs them as `DEFERRED_WITHDRAWAL` rather than failing the entire batch. Staff are shown a list of deferred copies with patron loan due dates so they can decide whether to wait for return or issue a recall notice. Once the item is returned, the system checks for a pending `DEFERRED_WITHDRAWAL` flag and routes the copy to the withdrawal queue instead of the reshelving queue. Forced withdrawal by a supervisor requires a two-step confirmation and generates an audit event with the override justification. |

## ISBN Validation Algorithm

ISBN validation is performed locally at the API layer before any external lookup is attempted, enforcing BR-15 (ISBN uniqueness enforced at `POST /catalog/items` and the import pipeline; duplicate merge requires staff confirmation). ISBN-10 validation computes a weighted checksum: each of the first nine digits is multiplied by its position weight (10 through 2), the products are summed, and the result modulo 11 must equal the check digit, where a remainder of 10 is represented as the character "X". ISBN-13 validation uses the GS1 algorithm: digits at odd positions (1-indexed) are multiplied by 1 and digits at even positions are multiplied by 3; the sum modulo 10, subtracted from 10 (with 10 treated as 0), must equal the final check digit. Both algorithms are applied client-side in the cataloging UI as a first-pass guard, and then re-validated server-side in the `CatalogItem` domain model before the database write.

When check-digit validation passes but the ISBN already exists in the `catalog_items` table, the API returns `409 Conflict` with `{"code":"DUPLICATE_ISBN","existingId":"<catalogItemId>"}`. This triggers the merge workflow: the cataloging UI opens a side-by-side comparison of the incoming record and the existing record, highlighting field-level differences. A staff member with the `CATALOGING` role must explicitly confirm which record becomes canonical and which is retired. The retirement step migrates all associated `BookCopy` rows, open holds, and loan history references to the canonical record before soft-deleting the duplicate and emitting a `catalog.item.merged` domain event. ISBN-10 and ISBN-13 representations of the same title are treated as equivalent by normalizing all ISBNs to ISBN-13 format (prefixing with "978" and recomputing the check digit) before uniqueness checks are performed.

## Metadata Enrichment Fallback Strategy

The metadata enrichment pipeline follows a strict priority order to maximize data completeness while tolerating external API failures: (1) local catalog — if a record with the same ISBN already exists, its metadata is reused without an external call; (2) OpenLibrary API — queried first because it provides richer DDC classification data and is fully open; (3) Google Books API — used as a fallback when OpenLibrary returns an error, empty result, or structurally invalid response; (4) manual staff entry — the final fallback when all automated sources are unavailable or return insufficient data.

Responses from both external APIs are cached in a Redis store with a TTL of 72 hours, keyed by normalized ISBN-13. This prevents redundant external calls during bulk imports and protects against API rate limits. Each API client enforces a 5-second connection timeout and a 10-second read timeout; failures increment a per-API circuit breaker counter. After five consecutive failures within a 60-second window, the circuit opens and the pipeline skips that API entirely for a 5-minute cooldown period, falling through to the next source in the priority order. If the enrichment pipeline is operating in degraded mode (one or both APIs circuit-broken), a `catalog.enrichment.degraded` health event is published to the monitoring system and the cataloging UI displays a non-blocking warning banner. Items created in `PENDING_METADATA` state are automatically retried against live APIs on a 30-minute schedule until all required fields are populated or a staff member performs manual entry.

## Bulk Import Rollback Strategy

The `POST /catalog/import` endpoint accepts a multipart CSV upload and processes records in transactional chunks of 50 rows. Each chunk is wrapped in a single database transaction: if any row within the chunk fails validation (invalid ISBN check digit, missing required fields, duplicate ISBN without staff override consent), the entire chunk is rolled back and the failure details are recorded in the import job's error log. Successfully committed chunks are not affected by a later chunk's failure, ensuring that a partial import always leaves the database in a consistent state with a known commit boundary.

The import job is persisted as a first-class entity with a `jobId`, `status` (`IN_PROGRESS`, `PARTIAL_FAILURE`, `COMPLETED`, `FAILED`), `totalRows`, `processedRows`, `failedRows`, `lastCommittedRow`, and a `failureDetails` array. Staff can query job status at any time via `GET /catalog/import/jobs/{jobId}`. When a job enters `PARTIAL_FAILURE` state, the UI surfaces a "Resume Import" action that re-submits the original file with a `resumeFromRow` parameter set to `lastCommittedRow + 1`, skipping already-committed records and avoiding duplicates. For catastrophic failures where data integrity cannot be guaranteed, staff with the `CATALOG_ADMIN` role can trigger a full rollback of the job by providing the `jobId` to `DELETE /catalog/import/jobs/{jobId}`, which issues compensating DELETEs for all `CatalogItem` and `BookCopy` records created by that job using the `importJobId` foreign key stored on each created record.
