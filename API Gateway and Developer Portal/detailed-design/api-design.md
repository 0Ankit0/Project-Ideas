# API Design — API Gateway and Developer Portal

## Overview

This document specifies the complete REST API surface of the API Gateway and Developer Portal system. All APIs conform to **OpenAPI 3.1** and follow the conventions described below.

### Design Principles

- **URI versioning** — every API is prefixed with a version segment (`/v1/`, `/v2/`). Breaking changes are introduced in a new version; non-breaking additions are made in-place.
- **Resource-oriented URLs** — URLs identify resources (nouns), HTTP methods identify operations (verbs).
- **JSON-first** — all request and response bodies are `application/json`. Binary uploads use `multipart/form-data`.
- **Consistent error envelope** — every error response uses the standard `ErrorResponse` schema defined in [Error Response Format](#error-response-format).
- **Idempotency keys** — mutating operations that must be safe to retry accept an optional `Idempotency-Key` request header.
- **Cursor-based pagination** — list endpoints use opaque cursor tokens rather than numeric offsets; see [Pagination](#pagination).
- **HTTPS everywhere** — all endpoints are served exclusively over TLS 1.2+.
- **UTC timestamps** — all datetime fields are ISO 8601 strings in UTC, e.g. `2024-03-15T10:30:00Z`.

### Versioning Strategy

| Version | Status | Notes |
|---|---|---|
| `/v1/` | **Stable** | Current production version |
| `/v2/` | Planned | Future breaking changes (response shape refactoring) |

Legacy version support window: **12 months** after a new major version is released.

## Authentication

Three authentication mechanisms are supported. Endpoints specify which mechanisms they accept.

### API Key (`X-API-Key`)

```
X-API-Key: gw_live_<base64url-encoded-32-bytes>
```

- API keys are generated by the platform and returned once at creation time.
- Keys are validated using HMAC-SHA256 prefix lookup + constant-time comparison.
- Scope: Gateway proxy endpoints and Management API (with appropriate key permissions).

### Bearer Token (JWT)

```
Authorization: Bearer <jwt>
```

- JWTs are issued by the OAuth 2.0 Auth Service (RS256 or ES256 signed).
- The gateway validates the signature, `exp`, `iss`, and `aud` claims.
- Short-lived access tokens (TTL 15 minutes) paired with refresh tokens (TTL 30 days).
- Scope: Developer Portal API and Management API.

### OAuth 2.0 Client Credentials

```
Authorization: Bearer <access_token>
```

- For server-to-server integration.
- Clients authenticate using `client_credentials` grant with `client_id` and `client_secret`.
- Access tokens carry `scope` claims that control endpoint access.

## Gateway Management API

The Gateway Management API (`https://api.example.com/mgmt`) is used by platform administrators and automation pipelines to configure gateway routes, consumers, API keys, and rate-limit plans.

**Authentication:** Bearer token with `admin` scope, or API key with `management` permission.

### Routes

#### `GET /v1/routes`

Lists all configured gateway routes.

**Query Parameters**

| Parameter | Type | Required | Description |
|---|---|---|---|
| `cursor` | string | No | Pagination cursor from a previous response |
| `limit` | integer | No | Page size (default: 20, max: 100) |
| `status` | string | No | Filter by status: `active`, `inactive`, `draft` |
| `tag` | string | No | Filter routes that have this tag |

**Response `200 OK`**

```json
{
  "data": [
    {
      "id": "rte_01HV8K2M3N4P5Q6R7S8T9U0V",
      "name": "Orders API v1",
      "path": "/v1/orders",
      "methods": ["GET", "POST"],
      "strip_path_prefix": "/v1/orders",
      "upstream_group_id": "ug_orders_prod",
      "status": "active",
      "tags": ["commerce", "critical"],
      "plugins_enabled": ["auth_api_key", "rate_limit_sliding", "transform_request"],
      "created_at": "2024-01-10T08:00:00Z",
      "updated_at": "2024-03-01T14:22:00Z"
    }
  ],
  "pagination": {
    "cursor": "eyJpZCI6InJ0ZV8wMUhWOEsyTTNONFA1UTZSNy",
    "has_more": true,
    "total_count": 47
  }
}
```

**Status Codes:** `200 OK`, `401 Unauthorized`, `403 Forbidden`

```bash
curl -X GET "https://api.example.com/mgmt/v1/routes?limit=20&status=active" \
  -H "Authorization: Bearer $MGMT_TOKEN"
```

## Developer Portal API

The Developer Portal API (`https://portal.example.com/api`) serves the self-service Developer Portal frontend. It is a BFF (Backend for Frontend) layer backed by Next.js Route Handlers.

**Authentication:** JWT Bearer token (session cookie managed by `next-auth`).

---

**Request Body**

## OAuth 2.0 Endpoints

The OAuth 2.0 authorization server is hosted at `https://auth.example.com`.

#### `POST /oauth/token`

Token endpoint supporting `client_credentials` and `authorization_code` grants.

**Content-Type:** `application/x-www-form-urlencoded`

**Client Credentials Grant — Request**

```
grant_type=client_credentials
&client_id=cli_01HV8K2M3N4P5Q6R7S8T9U0V
&client_secret=cs_abc123...
&scope=proxy:read proxy:write
```

**Authorization Code Grant — Request**

```
grant_type=authorization_code
&code=auth_code_abc123
&redirect_uri=https%3A%2F%2Fapp.example.com%2Fcallback
&client_id=cli_01HV8K2M3N4P5Q6R7S8T9U0V
&client_secret=cs_abc123...
&code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk
```

**Response `200 OK`**

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 900,
  "refresh_token": "rt_01HV8K2M3N4P5Q6R7S8T9U0V",
  "scope": "proxy:read proxy:write"
}
```

**Status Codes:** `200 OK`, `400 Bad Request` (invalid_request), `401 Unauthorized` (invalid_client), `403 Forbidden` (insufficient_scope)

#### `GET /oauth/authorize`

Authorization endpoint for `authorization_code` flow. Redirects the user to the login page.

**Query Parameters**

| Parameter | Required | Description |
|---|---|---|
| `response_type` | Yes | Must be `code` |
| `client_id` | Yes | OAuth client ID |
| `redirect_uri` | Yes | Must match registered redirect URI |
| `scope` | Yes | Space-separated scope list |
| `state` | Yes | CSRF protection value; returned unchanged |

## Error Response Format

All API errors use the following standard envelope regardless of the service.

### Error Object Schema

```json
{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "Request rate limit exceeded for consumer cns_01HV8.",
    "details": [
      {
        "field": null,
        "issue": "Quota of 1000 requests per minute exhausted. Retry after 12 seconds."
      }
    ],
    "trace_id": "01HV8K2M3N4P5Q6R7S8T9U0V",
    "docs_url": "https://docs.example.com/errors/RATE_LIMIT_EXCEEDED"
  }
}
```

### Error Codes Reference

| HTTP Status | Error Code | Description |
|---|---|---|
| `400` | `VALIDATION_ERROR` | Request body or query parameter failed validation. `details` contains per-field errors. |
| `401` | `UNAUTHORIZED` | Missing or invalid authentication credential. |
| `401` | `TOKEN_EXPIRED` | JWT or access token has expired. |
| `401` | `API_KEY_INVALID` | API key does not exist or has been revoked. |
| `403` | `FORBIDDEN` | Authenticated identity lacks required permission or scope. |
| `403` | `CONSUMER_SUSPENDED` | Consumer account has been suspended. |
| `404` | `NOT_FOUND` | Requested resource does not exist. |
| `409` | `CONFLICT` | Resource already exists (duplicate key, path, username). |
| `422` | `UNPROCESSABLE_ENTITY` | Request is syntactically valid but semantically incorrect. |
| `429` | `RATE_LIMIT_EXCEEDED` | Consumer has exceeded their request quota. |
| `500` | `INTERNAL_ERROR` | Unexpected server error. `trace_id` is provided for support. |
| `502` | `BAD_GATEWAY` | Upstream backend returned an invalid response. |
| `503` | `SERVICE_UNAVAILABLE` | Service temporarily unavailable (upstream down or circuit open). |
| `504` | `GATEWAY_TIMEOUT` | Upstream backend did not respond within the configured timeout. |

### Validation Error Detail

When `code` is `VALIDATION_ERROR`, the `details` array contains one entry per invalid field:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request body validation failed.",
    "details": [
      { "field": "email", "issue": "Must be a valid email address." },
      { "field": "password", "issue": "Must be at least 12 characters and contain at least one symbol." }
    ],
    "trace_id": "01HV8K2M3N4P5Q6R7S8T9U0V"
  }
}
```

## Rate Limiting Headers

All proxied and management API responses include the following headers to communicate quota state.

| Header | Type | Description |
|---|---|---|
| `X-RateLimit-Limit` | integer | Total requests allowed in the current window |
| `X-RateLimit-Remaining` | integer | Requests remaining in the current window |
| `X-RateLimit-Reset` | Unix timestamp (seconds) | Unix timestamp when the current window resets |
| `X-RateLimit-Window` | integer | Window duration in seconds |
| `Retry-After` | integer | Seconds to wait before retrying (present only on `429` responses) |
| `X-RateLimit-Policy` | string | Human-readable policy description, e.g. `1000;w=60` |

## Pagination

All list endpoints use **cursor-based pagination** to ensure consistent, performant results on large datasets regardless of concurrent insertions.

### Request Parameters

| Parameter | Type | Default | Description |
|---|---|---|---|
| `cursor` | string | — | Opaque cursor token from the previous page's `pagination.cursor` field. Omit for the first page. |
| `limit` | integer | `20` | Number of items per page. Maximum: `100`. |

### Response Pagination Object

```json
{
  "pagination": {
    "cursor": "eyJpZCI6ImNuc18wMUhWOEsyTTNONFA1UTZSNyIsImNyZWF0ZWRfYXQiOiIyMDI0LTAzLTE1VDEwOjMwOjAwWiJ9",
    "has_more": true,
    "total_count": 847
  }
}
```

| Field | Description |
|---|---|
| `cursor` | Opaque base64url-encoded token representing the position after the last returned item. `null` when there are no more pages. |
| `has_more` | `true` if there are more items beyond the current page. |
| `total_count` | Total number of items matching the query (not just the current page). May be approximate for very large datasets. |

### Pagination Example

**Page 1 — no cursor**

```bash
curl "https://api.example.com/mgmt/v1/consumers?limit=20" \
  -H "Authorization: Bearer $MGMT_TOKEN"
```

**Page 2 — use cursor from page 1 response**

```bash
curl "https://api.example.com/mgmt/v1/consumers?limit=20&cursor=eyJpZCI6ImNuc18wMUhW..." \
  -H "Authorization: Bearer $MGMT_TOKEN"
```

### Cursor Internals

Cursors are opaque base64url-encoded JSON objects containing the sort key(s) of the last returned item (e.g., `{"id": "cns_01HV...", "created_at": "2024-03-15T10:30:00Z"}`). Clients must treat cursors as opaque blobs — the internal structure is not part of the API contract and may change between versions.
