# Edge Cases: Operations

## Overview

This document covers critical operational edge cases for the API Gateway and Developer Portal
running on AWS ECS Fargate with RDS PostgreSQL 15, ElastiCache Redis 7, BullMQ, CloudFront, and
the OpenTelemetry observability stack. Operational failures differ from security or application-logic
failures in that they are primarily driven by infrastructure events: database failovers, rolling
deployments, scheduled maintenance, resource exhaustion, and configuration drift.

Operational edge cases frequently compound: a Redis memory exhaustion event degrades rate limiting
at the same time a deployment introduces a version mismatch, and the reduced observability from a
crashed OpenTelemetry collector makes both failures harder to diagnose. The scenarios below are
documented individually but each section notes where co-occurrence with other scenarios creates
elevated risk.

Infrastructure components covered:

- **AWS RDS PostgreSQL 15** — primary persistent data store for API keys, applications, analytics.
- **AWS ElastiCache Redis 7** — rate limit counters, session state, key validation cache.
- **AWS ECS Fargate** — container orchestration for gateway and portal services.
- **BullMQ on Redis** — asynchronous job processing for analytics, notifications, key propagation.
- **OpenTelemetry Collector** — distributed tracing pipeline feeding Jaeger and Prometheus.
- **AWS S3 + CloudFront** — static asset delivery and API documentation hosting.

### EC-OPS-001: RDS Primary Instance Fails Over to Standby Causing 30–60 Second Connection Disruption

| Field | Detail |
|-------|--------|
| **Failure Mode** | The RDS PostgreSQL 15 Multi-AZ primary instance becomes unavailable due to an underlying hardware failure, OS-level issue, or routine AWS maintenance event. RDS initiates an automatic failover to the standby replica in the secondary Availability Zone. The DNS CNAME for the RDS cluster endpoint is updated to point to the new primary. However, the ECS Fargate tasks running the gateway and portal maintain long-lived PostgreSQL connection pool connections (via `pg` Node.js client with `pg-pool`) that are still pointed at the old primary IP, which is no longer reachable. The connection pool keeps returning stale connections for the TCP keepalive timeout period (up to 60 seconds), during which all database queries fail with `ECONNRESET` or `ETIMEDOUT` errors. |
| **Impact** | During the 30–60 second failover window, all gateway operations that require a database read or write fail. This includes API key validation (if not cached in Redis), new OAuth token issuance, analytics writes, and all Developer Portal page loads. Consumers using uncached API keys receive HTTP 500 errors instead of their expected API responses. The Developer Portal returns HTTP 500 for all authenticated pages. After the DNS TTL expires and connection pools reconnect to the new primary, service recovers automatically without human intervention, but the outage window is a reliability metric regression. |
| **Detection** | AWS RDS emits a `RDS-EVENT-0006` (failover start) and `RDS-EVENT-0007` (failover complete) CloudWatch event. A CloudWatch Events rule triggers a PagerDuty alert within 30 seconds of the failover start event. Prometheus metric `pg_pool_error_rate` spikes from near-zero to 100% during the failover window, triggering an immediate alert. Grafana dashboard "Database Connection Health" shows the step-change in connection error rates and the subsequent recovery with a clear timestamp correlation to the RDS event. |
| **Mitigation / Recovery** | Configure `pg-pool` with aggressive reconnection settings: `idleTimeoutMillis: 10000`, `connectionTimeoutMillis: 3000`, and `allowExitOnIdle: false`. Implement a connection validation query (`SELECT 1`) on each connection before handing it to the application, with automatic pool flush (`pool.end()` then `new Pool()`) triggered by three consecutive connection errors within 5 seconds. The pool flush forces immediate DNS re-resolution, recovering connectivity as soon as the new primary is reachable (typically 5–10 seconds after the DNS CNAME update). After recovery, the BullMQ failed-job retry mechanism automatically re-processes any analytics or notification jobs that failed during the outage window. |
| **Prevention** | Set the RDS DNS CNAME TTL to 5 seconds (the minimum supported by Route 53 for RDS endpoints) to minimise the DNS propagation delay after failover. Enable `pg-pool` connection validation on checkout. Deploy a read replica in the same AZ as the primary ECS task cluster and direct all read-only queries (API key lookups, application fetches) to the read replica endpoint, so that read traffic is unaffected by primary failover entirely. Conduct a quarterly RDS failover drill in the staging environment to verify that the gateway recovers within the target RTO of 90 seconds and that the connection pool recovery logic functions correctly. |

### EC-OPS-002: ECS Task Definition Update Causes Brief Period with Mixed Old/New Gateway Versions Handling Traffic

| Field | Detail |
|-------|--------|
| **Failure Mode** | A gateway deployment updates the ECS task definition to a new Docker image (e.g., v2.5.0 → v2.6.0) with a database schema migration included. The ECS rolling update strategy is configured with `minimumHealthyPercent: 50` and `maximumPercent: 150`, meaning up to 50% of old tasks may continue running while new tasks come up. The database schema migration runs as part of the new task's startup sequence and adds a new required column to the `api_keys` table. Old task instances (v2.5.0) have no knowledge of this column and generate SQL queries that either fail on the new column constraint or silently omit the field. During the rolling window (typically 2–5 minutes), requests are load-balanced across old and new task versions, producing inconsistent API responses depending on which task handles the request. |
| **Impact** | During the mixed-version window, consumers experience non-deterministic behaviour: the same API request may succeed on one call and fail on the next depending on which ECS task handles it. If the schema migration adds a NOT NULL column without a default value, old tasks generate `null constraint violation` PostgreSQL errors on insert operations. New tasks correctly populate the new column. The inconsistency is particularly damaging for stateful operations (API key creation, OAuth token issuance) where a failed operation from an old task leaves partial state that a subsequent new task tries to read and fails on. |
| **Detection** | Grafana dashboard "Deployment Version Distribution" queries the custom `X-Gateway-Version` response header added by each task and shows the proportion of responses from each version in real time. A deployment-in-progress alert fires when more than one gateway version is detected simultaneously. Prometheus metric `gateway_error_rate_by_version` disaggregated by version label allows quick comparison of error rates between old and new task versions during the rolling window. Jaeger traces flagged with version labels expose which version generated each span in a distributed trace. |
| **Mitigation / Recovery** | Implement a blue-green deployment strategy instead of rolling updates for gateway releases that include database migrations: deploy the new task definition to a separate ECS service (green), run the migration against the database while the old service (blue) handles production traffic, then shift 100% of Route 53 weighted routing to the green service in a single atomic step. For already-in-progress rolling deployments that have caused errors, increase the deployment speed by setting `maximumPercent: 200` temporarily to accelerate replacement of old tasks, shortening the mixed-version window. Rollback is executed by updating the ECS service to the previous task definition revision. |
| **Prevention** | Enforce a "backward-compatible migration" policy: all database schema changes must be backward compatible with the currently deployed code version for at least one full deployment cycle. The pattern is: deploy v2.6.0 code that can handle both old and new schema → run additive migration (add nullable column with default) → deploy v2.7.0 code that uses the new column → run cleanup migration (add NOT NULL constraint). This eliminates the mixed-version incompatibility window entirely. Document this policy in the Engineering Deployment Runbook and enforce it in the CI/CD pipeline with a migration compatibility linter. |

### EC-OPS-003: Analytics PostgreSQL Partition Table Not Created Before Month Rollover Causes Insert Failures

| Field | Detail |
|-------|--------|
| **Failure Mode** | The `api_request_analytics` PostgreSQL table is range-partitioned by month (`PARTITION BY RANGE (created_at)`), with one child partition per calendar month (e.g., `api_request_analytics_2024_12`). A scheduled Lambda function is responsible for creating next month's partition on the 25th of each month. The Lambda function fails silently in December due to an IAM permission change that removed its `rds-data:ExecuteStatement` permission. On January 1st at 00:00:00 UTC, the first INSERT statement targeting the `api_request_analytics` table fails with PostgreSQL error `ERROR: no partition of relation "api_request_analytics" found for row` because the January partition does not exist. All subsequent analytics inserts fail for the entire month of January, causing a complete gap in usage data. |
| **Impact** | Analytics data for the entire affected month is lost if inserts fail without fallback. BullMQ analytics workers that process rate limit usage events accumulate a growing dead-letter queue of failed jobs. The Developer Portal shows zero usage for all API consumers for the month, creating consumer panic and support escalation. Billing systems cannot generate accurate invoices for the month. The gap in analytics data is an audit compliance failure for customers requiring continuous usage records for regulatory reporting. Recovery requires creating the partition and replaying all queued BullMQ events, which may take hours for a high-traffic month. |
| **Detection** | The Lambda function that creates partitions publishes a custom CloudWatch metric `partition_creation_success` set to 1 on success and 0 on failure. A CloudWatch alarm fires immediately when this metric is 0 after the expected execution time on the 25th of each month. A secondary safety net: a PostgreSQL cron job (via `pg_cron`) checks for the existence of next month's partition on the 28th of each month and sends an alert email if missing. The BullMQ dead-letter queue depth for the `analytics` queue firing a warning alert serves as a late-stage detection trigger. |
| **Mitigation / Recovery** | Immediately create the missing January partition via a direct PostgreSQL DDL statement: `CREATE TABLE api_request_analytics_2025_01 PARTITION OF api_request_analytics FOR VALUES FROM ('2025-01-01') TO ('2025-02-01')`. This statement can be executed with zero downtime while the table is online. Once the partition exists, failed BullMQ analytics jobs in the dead-letter queue are replayed sequentially using a dedicated recovery worker with `concurrency: 10` to avoid database overload. Verify the total event count after replay matches the expected count from Redis cumulative counters. Fix and redeploy the Lambda function with corrected IAM permissions and create partitions for the next 3 months proactively. |
| **Prevention** | Switch from a Lambda-based partition creation approach to a PostgreSQL `pg_partman` extension managed scheduled job running inside the RDS instance itself. `pg_partman` automatically creates future partitions on a configurable schedule (e.g., `premake: 3` creates the next 3 months' partitions in advance) and does not depend on external Lambda execution. Add an RDS instance-level CloudWatch alarm on `pg_partman_last_run` to detect if the extension's maintenance function has not run in more than 24 hours. Include a monthly partition existence check in the deployment smoke test suite that verifies the current month's and next month's partitions exist before marking a deployment successful. |

### EC-OPS-004: ElastiCache Redis Cluster Running at 95% Memory Causes Eviction of Rate Limit Counters

| Field | Detail |
|-------|--------|
| **Failure Mode** | The ElastiCache Redis 7 cluster reaches 95% memory utilisation due to a combination of growing BullMQ job payloads, accumulation of rate limit counter keys from a traffic spike, and OAuth session tokens not being evicted quickly enough. Redis's `maxmemory-policy` is configured as `allkeys-lru` (Least Recently Used eviction across all keys). Under memory pressure, Redis begins evicting keys, including active rate limit counter keys (`rl:{apiKey}:{window}`) that happen to have not been accessed recently due to a load-balancing pattern that routed recent traffic to a different Redis shard. With the rate limit counter keys evicted, the gateway recreates them on next request with a fresh counter starting at zero, effectively resetting the rate limit window for those API keys and allowing consumers to exceed their quotas. |
| **Impact** | Rate limit enforcement becomes unreliable for the affected API keys: their quotas effectively reset whenever memory pressure causes their counter keys to be evicted. High-traffic consumers whose keys are frequently accessed are less affected (LRU keeps hot keys); low-to-medium-traffic consumers whose keys sit idle between bursts are disproportionately affected. If the memory pressure persists for an extended period, the compounding effect of repeated quota resets enables significant quota overrun and backend overload. The rate limit system's integrity guarantee is violated without any visible error — the gateway continues returning HTTP 200 responses as if everything is working correctly. |
| **Detection** | CloudWatch metric `DatabaseMemoryUsagePercentage` for ElastiCache fires a warning alert at 80% and a critical alert at 90%. CloudWatch metric `CurrItems` vs. `EvictedKeys` provides a direct signal: Prometheus alert `redis_evicted_keys_rate` fires when `EvictedKeys` increases by more than 100 per second. Grafana dashboard "Redis Memory Health" shows the memory usage trend, eviction rate, and a correlation overlay with the rate-limit anomaly metric `rate_limit_counter_reset_early_total`, which spikes when eviction-driven resets are occurring. |
| **Mitigation / Recovery** | Immediately add ElastiCache cluster nodes to increase total cluster memory, restoring headroom. ElastiCache Online Scaling allows node addition without downtime. As a stopgap, change `maxmemory-policy` from `allkeys-lru` to `volatile-lru` combined with ensuring that rate limit counter keys (`rl:*`) have a TTL (they already should per EC-RATELIMIT-004 prevention) while BullMQ job keys and session tokens also have appropriate TTLs. With `volatile-lru`, only keys with TTLs are eligible for eviction, and since all keys should have TTLs, the eviction selection is bounded to expirable keys. Perform a manual audit to remove any keys without TTLs that are consuming memory unnecessarily. |
| **Prevention** | Size the ElastiCache cluster with a target steady-state memory utilisation of 60% to provide 40% headroom for traffic spikes. Configure a CloudWatch alarm at 70% to trigger an automatic scaling event (adding a read replica) before reaching the eviction threshold. Separate rate limit counter data into a dedicated Redis logical database (DB 0) and configure BullMQ to use a separate Redis instance or logical database (DB 1), preventing BullMQ memory growth from competing with rate limit keys. Conduct a monthly Redis memory audit using `redis-cli --bigkeys` and `MEMORY USAGE` to identify oversized keys and implement TTL corrections. |

### EC-OPS-005: OpenTelemetry Collector Crashes Causing Distributed Trace Data Loss for 15 Minutes

| Field | Detail |
|-------|--------|
| **Failure Mode** | The OpenTelemetry (OTel) Collector sidecar container deployed in each ECS task group crashes due to an OOM kill caused by a trace data burst (e.g., a traffic spike generating 10× normal span volume). The ECS service restarts the sidecar container, but the restart takes 60–90 seconds. During the downtime window, gateway ECS tasks continue generating trace spans and exporting them to the crashed collector via the OTLP gRPC endpoint (`localhost:4317`). Because the collector is unavailable, the OTel SDK's gRPC export calls fail. The SDK's retry queue (bounded at 512MB in-memory) fills up and begins dropping spans after 2 minutes. The collector restarts after 90 seconds but cannot accept the batched retries faster than they are generated; the queue drains slowly over 15 minutes while new spans continue arriving. Net result: approximately 15 minutes of distributed trace data is partially or fully lost. |
| **Impact** | Distributed trace data for the 15-minute outage window is unavailable in Jaeger. If a production incident occurs during or shortly after this window — for example, a latency regression or error spike — engineers cannot use distributed traces to identify the root cause, extending the mean time to identify (MTTI) and mean time to resolve (MTTR) for the incident. Prometheus metrics and CloudWatch logs are unaffected (separate pipelines), so aggregate metrics remain available, but the per-request trace context needed for granular debugging is missing. For SLA reporting purposes, the trace data gap creates a hole in the evidence trail. |
| **Detection** | The OTel Collector exposes a Prometheus endpoint at `localhost:8888/metrics`. The metric `otelcol_exporter_send_failed_spans_total` increments when spans fail to export to the Jaeger backend. A Prometheus alert fires when this counter increases by more than 1,000 in a 60-second window. The ECS container health check for the OTel sidecar reports unhealthy status within 30 seconds of the crash; CloudWatch Container Insights `ECS/ContainerInstanceCount` metric reflects the restart. PagerDuty alert "OTel Collector Restart" is sent when the ECS sidecar restart count exceeds 2 in a 10-minute window. |
| **Mitigation / Recovery** | Configure the OTel Collector's memory limiter processor with `check_interval: 1s`, `limit_mib: 400`, and `spike_limit_mib: 100` to prevent OOM kills by applying backpressure to span exporters before memory is exhausted. Increase the ECS sidecar container's memory reservation from the default to 512MiB to accommodate legitimate bursts. After the collector restarts, triage the in-memory retry queue by temporarily increasing the Jaeger exporter's `sending_queue.num_consumers` from 10 to 50 to accelerate drain. For the data loss window, correlate available Prometheus metrics and CloudWatch logs to reconstruct the incident timeline without trace data. |
| **Prevention** | Deploy the OTel Collector as a standalone ECS service (separate from the gateway sidecar) with its own ECS Fargate task, sized with 2GB memory and auto-scaling based on the `otelcol_receiver_accepted_spans` metric. Using a standalone collector decouples the collector's stability from individual gateway task restarts and allows independent scaling. Configure the OTel SDK on gateway tasks to use a persistent gRPC retry with exponential backoff and a maximum retry duration of 5 minutes, spanning typical collector restart times. Enable OTel Collector's `file_storage` extension to persist the retry queue to an EFS volume, surviving container restarts without data loss. |

### EC-OPS-006: S3 Bucket Policy Misconfiguration Accidentally Makes API Documentation Publicly Writable

| Field | Detail |
|-------|--------|
| **Failure Mode** | A Terraform change intended to allow CloudFront Origin Access Control (OAC) to read from the documentation S3 bucket accidentally sets the bucket policy principal to `"*"` (all principals) instead of the OAC service principal ARN. The policy grants `s3:PutObject` and `s3:DeleteObject` permissions to the wildcard principal, making the bucket publicly writable. This is deployed to production via a Terraform `apply` in an automated deployment pipeline that does not include a human-review gate for S3 policy changes. An automated scanner or a malicious actor discovers the open bucket within minutes using S3 bucket policy scanning tools or via AWS public access block status APIs. |
| **Impact** | Anyone on the internet can overwrite or delete files in the documentation S3 bucket. In the worst case, an attacker replaces the API documentation HTML, JavaScript bundles, or OpenAPI spec files with malicious content (supply chain attack via documentation), potentially injecting XSS payloads that execute in developers' browsers when they read the docs. The CloudFront CDN serves the tampered content globally with up to 24-hour cache TTLs (EC-OPS-008 is a compounding factor here), meaning the malicious content is immediately distributed and cached at all 400+ CloudFront edge locations. Restoring legitimate content requires both an S3 fix and a global CloudFront cache invalidation. |
| **Detection** | AWS Config rule `s3-bucket-public-write-prohibited` evaluates all S3 bucket policies continuously and fires a non-compliant finding within 5 minutes of the policy change. AWS Security Hub aggregates this finding as a HIGH severity alert. CloudTrail logs the `PutBucketPolicy` API call with the full policy document, enabling precise attribution of the change. An S3 Server Access Log anomaly detector (Lambda function triggered by CloudTrail) alerts when `PutObject` calls originate from principals outside the expected OAC service principal ARN list. |
| **Mitigation / Recovery** | Immediately revert the S3 bucket policy to the correct OAC-scoped read-only policy using `aws s3api put-bucket-policy`. Enable S3 Block Public Access at both the bucket and account level: `aws s3api put-public-access-block --public-access-block-configuration BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true`. Issue a CloudFront cache invalidation for `/*` to purge any potentially tampered content from all edge locations. Restore the documentation files from the most recent S3 versioning snapshot (S3 versioning must be enabled — verify this is configured). Audit S3 access logs from the exposure window for any unexpected `PutObject` or `DeleteObject` calls and verify file integrity by comparing S3 object ETags against the known-good deployment artifacts in the CI/CD pipeline. |
| **Prevention** | Add a required human approval step in the CI/CD pipeline (GitHub Actions environment with protection rule) for any Terraform plan that includes changes to S3 bucket policies, IAM policies, or security group rules. Enable AWS S3 Block Public Access at the AWS account level as a guardrail that overrides any accidental wildcard bucket policy. Use `terraform-compliance` or Open Policy Agent (OPA) policy checks in the Terraform pipeline that fail the plan if any S3 bucket policy contains `"Principal": "*"` combined with write actions. Enable S3 object versioning and S3 Object Lock (compliance mode) on the documentation bucket to ensure file integrity can always be restored. |

### EC-OPS-007: BullMQ Worker Crashes with Jobs in Active State Leaving Orphaned Jobs Never Completed

| Field | Detail |
|-------|--------|
| **Failure Mode** | A BullMQ worker ECS task processing a batch of analytics jobs crashes mid-execution (OOM kill, unhandled exception, or ECS task health-check failure). BullMQ marks jobs as `active` when a worker picks them up by moving them from the `wait` list to the `active` sorted set in Redis. When the worker crashes without calling `job.moveToCompleted()` or `job.moveToFailed()`, the jobs remain in the `active` state indefinitely. BullMQ's stalled job detection mechanism (StalledJobsCheck) is disabled or configured with an excessively long interval (e.g., 30 minutes), so the orphaned active jobs are not automatically retried. They sit in the `active` set consuming queue slot capacity but never being processed. |
| **Impact** | Orphaned active jobs represent unprocessed work that accumulates silently. For analytics jobs, this means quota usage events are never written to PostgreSQL, causing billing and usage reporting gaps (similar to EC-RATELIMIT-006 but with a different root cause). For key-propagation jobs, new API keys remain unactivated in Redis, blocking developers from using them. For email notification jobs, account-related emails (plan change confirmations, security alerts) are never sent. The maximum impact depends on the job types orphaned; key-propagation and security-notification jobs have the highest consumer-facing severity. |
| **Detection** | Prometheus metric `bullmq_active_jobs_age_seconds_max` (maximum age of any job currently in `active` state) fires an alert when any active job has been in that state for more than twice its expected processing duration (e.g., alert when an analytics job has been active for more than 60 seconds when the expected processing time is 5 seconds). BullMQ's built-in `getJobCountByTypes()` API is polled every 30 seconds by a monitoring Lambda; a sudden increase in `active` count without a corresponding increase in `completed` count indicates orphaned jobs. CloudWatch alarm `BullMQ_OrphanedActiveJobs` fires when active count exceeds 100 for more than 5 consecutive minutes. |
| **Mitigation / Recovery** | Enable BullMQ's StalledJobsCheck with `stalledInterval: 30000` (30 seconds) and `maxStalledCount: 3`. This automatically moves stalled jobs back to the `wait` state after they exceed the stalled interval, making them available for processing by other healthy workers. Restart the crashed ECS task and confirm worker health. Run a one-time BullMQ job recovery script: `queue.getJobs(['active'])` filtered by `processedOn < Date.now() - 120000` (active for more than 2 minutes), then call `job.retry()` on each to manually requeueing them. Verify job completion counts return to expected processing rates after recovery. |
| **Prevention** | Enable BullMQ StalledJobsCheck in every worker configuration with `stalledInterval: 30000` and `maxStalledCount: 2`. Use BullMQ's `lockDuration` option set to 30 seconds: if a worker holds a job lock for longer than `lockDuration`, BullMQ automatically reclaims the job for another worker. Implement a graceful shutdown handler in the ECS worker service: on SIGTERM (sent by ECS during task replacement), the worker calls `worker.close(true)` (force close) which causes BullMQ to release all active job locks immediately, making them available for immediate pickup by other workers rather than waiting for the stalled job timeout. Deploy at minimum 2 worker ECS tasks at all times to ensure a surviving worker can reclaim jobs from a crashing peer. |

### EC-OPS-008: CloudFront Cache Serves Stale Gateway Configuration to Developer Portal for Up to 24 Hours

| Field | Detail |
|-------|--------|
| **Failure Mode** | The Developer Portal's Next.js application fetches gateway configuration data (available API endpoints, plan tier limits, supported OAuth scopes) from a JSON configuration API endpoint (`/api/config/gateway`) behind CloudFront. This endpoint has a `Cache-Control: public, max-age=86400` header (24-hour cache) set to reduce origin load. When the gateway team updates the configuration — for example, adding a new API product, changing rate limit values for a plan tier, or deprecating an OAuth scope — the updated configuration is served from the origin immediately, but CloudFront continues serving the stale 24-hour-old cached response to all portal users and to the Next.js server-side rendering cache. Developers see incorrect plan limits, missing API products, or stale OAuth scope lists in the portal UI for up to 24 hours after the change. |
| **Impact** | Developers trying to subscribe to a newly launched API product do not see it in the portal's product catalogue for up to 24 hours, directly delaying adoption and revenue. Developers viewing their plan limits see incorrect quota values, causing confusion and misdirected support tickets ("my plan says 1,000 req/min but I'm getting 429 errors at 500 req/min"). If a deprecated OAuth scope is still shown as available, developers build integrations against it, only to discover their tokens are rejected when the scope is finally removed. For enterprise sales cycles, a 24-hour delay in configuration visibility is a significant customer experience failure. |
| **Detection** | A post-deployment Lambda function compares the CloudFront-served response body for the configuration endpoint against the expected new configuration hash (stored in AWS SSM Parameter Store as part of the deployment pipeline). If the hashes differ after the deployment's CloudFront invalidation step completes, the Lambda emits a `cloudfront_stale_config_detected` CloudWatch metric and sends a Slack alert to the engineering channel. Grafana dashboard "Portal Configuration Freshness" tracks the age of the configuration response by comparing the `Last-Modified` response header against the latest deployment timestamp. |
| **Mitigation / Recovery** | Issue an immediate CloudFront cache invalidation for the configuration endpoint path: `aws cloudfront create-invalidation --distribution-id EDFDVBD6EXAMPLE --paths "/api/config/gateway"`. CloudFront invalidations typically propagate to all edge locations within 60 seconds. After the invalidation, verify the portal serves the updated configuration by fetching the endpoint from three geographically distributed locations (use `curl` from different AWS regions via a Lambda). If the portal UI caches the configuration in client-side state (Next.js `SWR` or `React Query`), publish a portal version bump that forces client-side cache revalidation via a changed `Cache-Control: no-cache` response header or a version query parameter change. |
| **Prevention** | Include a CloudFront cache invalidation step for all affected API paths as a mandatory post-deployment step in the CI/CD pipeline. Use `Cache-Control: public, max-age=300, stale-while-revalidate=60` (5-minute cache, 1-minute stale-while-revalidate) instead of 24-hour caching for configuration endpoints, limiting the maximum staleness to 6 minutes while still reducing origin load significantly. Implement cache-busting via a `?v={deploymentHash}` query parameter in the portal's configuration fetch URL, so that each deployment automatically bypasses the existing cache entry. Add a smoke test to the deployment pipeline that verifies the CloudFront-served configuration matches the expected configuration within 2 minutes of deployment completion. |
